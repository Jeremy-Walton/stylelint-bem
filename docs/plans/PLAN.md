# stylelint-bem — Implementation Plan

A stylelint plugin that validates BEM methodology in CSS, using native nesting. Rules apply only to selectors that look like BEM (contain configured separators), not every selector.

## Decisions (from interview, 2026-07-21)

| Topic | Decision |
|---|---|
| Language | TypeScript, compiled for publishing |
| Stylelint | ~~16.x only (ESM)~~ **Widened 2026-07-21:** `^16.0.0 \|\| ^17.0.0`. A real run against stylelint 17.10.0 (see the real-project dogfood in Handoff) worked with no changes; not yet cross-checked against stylelint's own 16→17 migration notes, so treat as "observed to work once," not "verified compatible." |
| BEM detection | Selector contains element/modifier separator; ignore list configurable |
| Name grammar | Parts must be kebab-case; separator-containing classes with non-kebab parts are invalid BEM, flagged by new `valid-name` rule (decided 2026-07-21) |

| Orphan scope | ~~Parent `.block` must be defined in the same file~~ **Superseded 2026-07-21:** project-wide — a block defined anywhere in the project satisfies the check, not just the current file. | Autofix | None — report only |
| Tests | Vitest + stylelint's testRule-style harness; strict TDD (see below) |
| Distribution | Publish to npm; also ship a recommended shareable config |
| Package name | `stylelint-bem` (confirmed) |
| Rule namespace | ~~`plugin/stylelint-bem-*` (one rule per check)~~ **Superseded 2026-07-21:** a single rule, `plugin/stylelint-bem`, whose primary option is an options object (mirrors `stylelint-selector-bem-pattern`'s style). Each former "rule" becomes a `checks.<name>` boolean, opt-out (all default `true`). Reason: stylelint never runs a rule that isn't a key in `config.rules`, so per-check rules are inherently opt-in; folding checks into one rule's options is the only way to make them opt-out by default. |
| Project-wide scope mechanism | (decided 2026-07-21, interview) Glob-scan the disk: the plugin does its own file discovery and parses matched files with `postcss.parse()`, independent of whatever subset of files stylelint is linting this invocation — not invocation-scoped (would need a fragile `Processor`-based mechanism). Confirmed feasible: stylelint rules may return `Promise<void>` (`RuleBase` type), so async file I/O isn't blocked. |
| Project-wide file discovery | (decided 2026-07-21, interview) Auto-infer the project root by walking upward from the linted file to the nearest `package.json`; glob `**/*.css` under that root, excluding `node_modules`. No required option for the common case. |
| Third-party/external blocks | (decided 2026-07-21, interview) New `knownBlocks: string[]` rule option — block names always treated as defined, wherever they appear (root block, or an element/modifier's parent), regardless of selector shape. Use case: vendor component classes (e.g. `react-select__*`) that will never be defined in any project CSS file. Chosen over reusing `ignoreSelectors`, which is selector-shaped and would need a separate pattern per compound-selector variant. |
| No `package.json` found | (decided 2026-07-21, interview) Fall back to same-file-only behavior (today's Phase 1 behavior) rather than erroring or scanning from some other root guess. |
| Monorepo root boundary | (decided 2026-07-21, interview) Stop at the **nearest** `package.json` walking up from the linted file — package-scoped, not workspace-root-scoped. A block shared across sibling packages in a monorepo is out of scope for auto-discovery (same trade-off as same-file-only was for cross-file within a single package; can revisit if it proves noisy in practice). |
| Symlinks | (decided 2026-07-21, interview) Do not glob through symlinked files/directories. A symlinked package (e.g. a linked design-system dependency) is treated as an external dependency, not part of the project being scanned — same category as `node_modules`. |

## TDD workflow (applies to every phase)

Each phase is red → green → refactor:

1. **Red:** write the phase's tests first (valid/invalid fixtures, expected messages). Run them and confirm they fail for the right reason (rule missing/unimplemented — not a harness error).
2. **Green:** implement the minimum to make them pass. No behavior without a driving test.
3. **Refactor:** clean up with the suite green.

Each phase's PAUSE happens at green — reviewer adjustments become new failing tests first, then fixes.

## Phase 1.5 — Project-wide orphan scope & known-blocks allowlist — COMPLETE (2026-07-21)

Supersedes the same-file-only orphan scope decided in Phase 1, per the real-world dogfood (see Handoff). Mechanism decisions recorded in the Decisions table above (glob-scan the disk, auto-infer project root, `knownBlocks` option; edge cases: no-`package.json` → same-file fallback, monorepo → nearest package, symlinks not followed).

**Implemented, red → green throughout:**
- `src/utils/project-scan.ts` (new): `findProjectRoot(startDir)` walks upward via `fs.existsSync` looking for the nearest `package.json`, returning `null` if none is found before hitting the filesystem root. `scanProjectDefinedClasses(projectRoot)` globs `**/*.css` under the root via `fast-glob` (`ignore: ['**/node_modules/**']`, `followSymbolicLinks: false`), parses each match with `postcss.parse()`, and unions the results of `buildDefinedClassIndex` per file — parse failures on individual files are caught and skipped, not fatal. Results are cached in a module-level `Map` keyed by project root for the process lifetime (no invalidation yet — acceptable per the original open-question note; revisit only if it proves to be a real problem in a long-running watch/LSP process). `scanProjectDefinedClassesForFile(root)` is the composition the rule calls: reads `root.source?.input.file` (undefined for `stylelint.lint({ code })` calls with no `from`/`codeFilename` — confirmed via a quick postcss probe, not a guess) to decide whether there's a real file to walk up from at all; returns an empty set immediately if not, which is the same-file-only fallback.
- `src/utils/rule-options.ts`: added `knownBlocks?: string[]` to `BemSharedOptions`/`sharedOptionsSchema`, plus `resolveKnownBlocks()` (mirrors `resolveSeparatorOptions`).
- `src/rules/stylelint-bem/check-context.ts`: `CheckContext` gained a `knownBlocks: Set<string>` field, plus a new `isDefinedOrKnown(context, block, targetClassName)` helper used by both orphan checks — `knownBlocks.has(block)` short-circuits before falling back to `definedClassIndex.has(targetClassName)`, so a known block satisfies the check regardless of what the immediate target string looks like (root block or an element's target alike).
- `src/rules/stylelint-bem/index.ts`: the rule function is now `async` (confirmed stylelint's `RuleBase` type permits `Promise<void>`, so this isn't a compatibility break). `definedClassIndex` is now `new Set([...projectClasses, ...buildDefinedClassIndex(root)])` — a plain union, not an explicit "current file wins" merge: the current file is *never excluded* from the disk scan (excluding it would key the module-level cache per-linted-file and defeat caching entirely), so the in-memory `buildDefinedClassIndex(root)` result is unioned on top and dominates simply by being additive; a stale on-disk copy of the same file can only ever add classes, never hide ones the in-memory AST has. Verified directly by a test that lints an in-memory `code` string via `codeFilename` pointing at a path that's never written to disk at all.
- No new dependency install weight: `fast-glob` was already present transitively (via stylelint's own deps); promoted to a direct dependency since it's now imported directly. `postcss` moved from `devDependencies` to `dependencies` for the same reason (previously only used via type-only imports, now called at runtime in `project-scan.ts`).
- Tests: `src/utils/project-scan.test.ts` (12, real `fs.mkdtemp` fixtures, no mocks — including a black-box caching test that mutates the directory between two calls to the same root and asserts the second call still returns the stale/cached result), `knownBlocks` fixtures added to both orphan checks' test files, and a new "project-wide orphan scope" describe block in `src/rules/stylelint-bem/index.test.ts` (4 tests: cross-file resolution, still-orphaned-if-truly-undefined, in-memory-wins-over-disk via `codeFilename`, no-file-path fallback).

## Phase 2 — Grammar checks (name shape) — COMPLETE (2026-07-21)

**Implemented, red → green throughout:**
- `src/rules/stylelint-bem/checks/valid-name.ts` (`checkValidName`): for every BEM-shaped class, `isKebabCase(parsed.block) && parsed.segments.every(s => isKebabCase(s.name))` — one violation per offending class (not per bad part), reported via the new `invalidName` message. Non-BEM classes (no separators) are never checked, matching the project's core premise.
- `src/rules/stylelint-bem/checks/no-double-nested-element.ts` (`checkNoDoubleNestedElement`): scans `parsed.segments` from index 1 onward for the first `element`-type segment; if found, the segment immediately before it decides the message — `element` before it → `doubleNestedElement` (with a flatten suggestion: `formatClassName(block, [lastSegment], separatorOptions)`, i.e. block + the *last* segment in the chain, dropping everything in between, matching `CHECKS.md`'s own `.block__element__other` → `.block__other` example); `modifier` before it → `elementAfterModifier` (no flatten suggestion — there's no single "correct" fix to suggest). Only the first violation in a class is reported.
- **Check-context refactor to support this:** `CheckContext.message: RuleMessage` (one message per check) became `CheckContext.messages: Record<string, RuleMessage>` (the full `messages` bag, built once), and `reportBemViolation` now takes an explicit `message` argument instead of reading `context.message`. This was needed because `noDoubleNestedElement` is one check with two distinct message templates depending on violation subtype — a single message-per-check field couldn't express that. Side benefit: `index.ts`'s dispatch loop simplified further — `context` is now built once and reused identically for every check (no more per-check `{ ...context, message }` spread), and `CHECK_DEFINITIONS` is just `Record<CheckName, CheckRunner>` (no `message` field in the registry at all).
- **Existing-test isolation fixes** (both new checks default `true`, so every other check's fixture set needed auditing for accidental overlap): `.card--featured__title` (the "modifier-then-element is invalid but not this check's concern" accept case, present in both `no-orphaned-element.test.ts` and `no-orphaned-modifier.test.ts`) and `.card__header__title` (the double-nested reject case in `no-orphaned-element.test.ts`) now also disable `noDoubleNestedElement` in their `checks` config, so each test file still isolates the one check it's actually testing.
- **Tricky-name regression fixture note:** a fixture using a literal single underscore (`.card--is_active`) to prove the parser doesn't mistake it for a chained separator also tripped `validName` (underscores aren't valid kebab-case characters) — fixed by disabling `validName` in that specific test block, since the fixture's purpose is purely to test `noDoubleNestedElement`'s parsing, not name validity.
- **Custom-separator test fixture note:** an earlier draft used `~~` as a custom `modifierSeparator` in a CSS fixture; `~` is a live CSS combinator character to `postcss-selector-parser`, so `.card~~featured-title` didn't parse as a single class at all. Switched to `_` for that test.

## Phase 3 — Nesting check (the big one) — COMPLETE (2026-07-21)

- **Red:** extensive fixtures for the `requireNesting` check, written and failing before any implementation:
  - `.block__el` must appear as a full selector nested (at any depth) inside a `.block` rule.
  - `.block--mod` must appear as `&.block--mod` compound directly under `.block`.
  - `.block__el--mod` must appear as `&.block__el--mod` under `.block__el` (which itself is nested in `.block`).
  - Selector lists, at-rules (`@media` inside blocks), deeper nesting, combinators.
- **Green:** implement, incrementally driving each fixture group.

**Acceptance:** all listed shapes covered by tests that went red → green; behavior inside `@media`/`@supports` decided and tested.

**Implemented, red → green throughout:**
- **`@media`/`@supports` decision (interview, 2026-07-21):** at-rules are transparent — they never count as a nesting level. A modifier compound-nested directly inside a `@media` that is itself directly inside its block still satisfies "directly under"; an element inside a `@media` at any depth inside its block still satisfies "at any depth". Implemented via `src/utils/rule-ancestors.ts`'s `findAncestorRules(node)`, which walks `node.parent` collecting only `type === 'rule'` ancestors — `atrule`/`root` nodes are skipped, not counted, but also never block the walk from continuing further up.
- **Core algorithm** (`src/rules/stylelint-bem/checks/require-nesting.ts`, `checkRequireNesting`): for every BEM class occurrence found by `forEachBemClass`, `expectedParentName = formatClassName(block, segments.slice(0, -1), separatorOptions)` — the same "immediate target minus last segment" formula `noOrphanedModifier` already used. The **last segment's type alone** decides which branch runs (no special-casing for malformed multi-segment shapes like `.block--mod__el` — they fall through the same logic and get flagged as usual, since satisfying `requireNesting` for them is impossible by construction):
  - Last segment `modifier`: the occurrence's own selector must be `'ampersand'` shape (`&.class`, `modifierNotCompound` if not) AND the class's *immediate* ancestor rule (`findAncestorRules(ruleNode)[0]`, i.e. nearest, at-rules skipped) must have a `'bare'`-shaped selector matching `expectedParentName` (`modifierNotNestedDirectly` if not, or if there's no ancestor at all).
  - Last segment `element`: the occurrence's own selector must be `'bare'` shape (plain `.class`, not compounded with `&`; `elementNotFullSelector` if not) AND **any** ancestor rule (`findAncestorRules(ruleNode).some(...)`, any depth) must have a `'bare'`-shaped selector matching `expectedParentName` (`elementNotNested` if none do).
- **Selector shape classification** (`src/utils/selector-walker.ts`, new `getClassNodesWithNestingShape`/`NestingShape`/`ClassNodeWithShape` — kept separate from the existing `getClassNodes`/`ClassNode` rather than extending them in place, so the exact-object-literal assertions in `selector-walker.test.ts` for the other four checks didn't need touching): for a given class node, walks backward/forward through its *immediate* container's sibling nodes to find the compound (run of simple selectors with no combinator boundary) it belongs to. `'bare'` — compound is the selector's leading (first) compound, contains only this class (pseudo-classes/attributes tolerated alongside it), no `&`. `'ampersand'` — same, but the compound also contains a nesting (`&`) node. `'other'` — anything else: preceded by a combinator (not leading), compounded with another class/tag/id/universal, or not found. Works recursively for classes inside `:is()`/`:not()` params too, since it operates on the class node's actual parent container rather than assuming top-level.
- **Ancestor walking** (`src/utils/rule-ancestors.ts`, new file, not folded into `selector-walker.ts` since it walks the postcss *tree* — `Rule.parent` — rather than parsing a selector *string*): `findAncestorRules(node: Rule): Rule[]` returns every `Rule` ancestor from nearest to furthest, silently skipping `AtRule` wrappers (the `@media` transparency decision) and stopping at `root`.
- **`forEachBemClass` gained a 4th callback argument, `selector: string`** (`check-context.ts`) — the specific selector string (from `ruleNode.selectors`) the visited class node came from. Needed because `requireNesting` must classify shape per-occurrence (a class can appear with different shapes across a selector list), and the existing three checks' callbacks don't declare a 4th parameter, so this is source-compatible (TS structurally allows a narrower-arity callback to satisfy a wider-arity parameter type) — no changes needed to the other three checks.
- **Existing-test isolation** (same pattern as Phase 2's `noDoubleNestedElement` audit, larger in scope this time since almost every existing fixture across the suite is a deliberately *flat*, non-nested selector to isolate its own check): added `requireNesting: false` to the `checks` config of every `testRule` block in `no-orphaned-element.test.ts`, `no-orphaned-modifier.test.ts`, `no-double-nested-element.test.ts`, `valid-name.test.ts`, and most of `index.test.ts`/`configs/recommended.test.ts`/top-level `index.test.ts` (the handful of fixtures that were already properly nested — e.g. the "accepts a fully-defined block/element/modifier chain with both checks on" test — were left using the bare `true`/no-override config, since they don't need isolating). `configs/recommended.test.ts`'s "flags orphaned elements and modifiers out of the box" test now expects 4 warnings, not 2 (the flat fixture trips both an orphan check and `requireNesting` per class); top-level `index.test.ts`'s "registers the rule..." test now expects two `ruleName` warnings, not one, for the same reason.
- **Observed relationship between `requireNesting` and the orphan checks (not a bug, just a note for future readers):** satisfying `requireNesting` for a class structurally requires a real ancestor `Rule` whose selector is its target block/element — and that ancestor rule, by existing at all in the same file, always also satisfies the orphan check's "is it defined anywhere" test. So a class can never fail an orphan check while passing `requireNesting`; the interesting/common real-world case (per the Phase 1.5 dogfood) is the reverse — passing the (project-wide) orphan check while failing `requireNesting`, e.g. a modifier applied to a block defined in a different file, or applied flat/top-level in the same file.
- New messages: `elementNotFullSelector`, `elementNotNested`, `modifierNotCompound`, `modifierNotNestedDirectly` — added to `src/rules/stylelint-bem/index.ts`'s `messages` map and registered as `requireNesting` in `CHECK_DEFINITIONS`.
- Tests: `tests/rules/stylelint-bem/checks/require-nesting.test.ts` (25 new tests — accept/reject for elements, modifiers, and element-modifiers; selector lists; `@media` transparency for both a valid nested case and an invalid unrelated-placement case; pseudo-classes attached to the BEM class; a combinator-preceded case; an intermediate-wrapper-breaks-direct-nesting case for modifiers; a cascading-isolation case showing a malformed element doesn't spuriously fail its otherwise-valid modifier; `ignoreSelectors`; custom separators).

**Acceptance:** all green — 154 tests total (was 129 before this phase), `tsc --noEmit` clean, `npm run build` clean.

**Simplification pass (2026-07-21, `/simplify`):** four parallel review agents (reuse/simplification/efficiency/altitude) found the same root cause from three independent angles — `getClassNodesWithNestingShape` (added this phase) was a near-duplicate of the existing `getClassNodes`, causing every selector to be parsed twice per BEM class occurrence (once by `forEachBemClass` to find the class node, again by `require-nesting.ts`'s `shapeOf` to classify it), and `forEachBemClass`'s `visit` callback had grown a `selector: string` 4th argument used by only one of five checks, solely to enable that second parse. Simplification agent found the diff otherwise clean.
- **Fixed:** merged the two functions — `getClassNodes`/`ClassNode` now carries `nestingShape` directly (computed once, during the single parse), `getClassNodesWithNestingShape`/`ClassNodeWithShape` removed. `forEachBemClass`'s `visit` callback dropped back to 3 arguments (`ruleNode`, `classNode`, `parsed`) — no selector re-threading needed. `require-nesting.ts` reads `classNode.nestingShape` directly instead of re-parsing via a `shapeOf` helper (removed). Updated `selector-walker.test.ts`'s pre-existing exact-`toEqual` assertions on `getClassNodes` to include `nestingShape` (small blast radius — one production caller, one test file — so updating beat keeping the types parallel), and added direct unit coverage for shape classification (`&`-compound, trailing pseudo-classes, descendant combinators, `:is()`) that previously only existed indirectly via `require-nesting.test.ts`'s integration-level fixtures. 159 tests green (was 154), `tsc`/build clean.
- **Skipped:** the efficiency agent's secondary suggestion to memoize `ruleHasBareMatch`'s per-`Rule` parse (e.g. a `WeakMap<Rule, ClassNode[]>`) so sibling elements/modifiers under the same parent don't each re-parse that parent's selectors — real but, per the agent's own assessment, negligible for hand-authored CSS file sizes; adding a cache here would be optimizing a cold path at the cost of readability. Revisit only if `requireNesting` is ever run interactively (e.g. LSP-on-keystroke) against very large generated stylesheets.

This matches `CHECKS.md`'s literal wording (`knownBlocks` is documented only under the orphan checks), so it's working as documented — just a sharper edge than it might look at first glance, worth being aware of for the same reason as above.
